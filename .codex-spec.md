## What changed in this updated version

This update corrects four important document defects from the original draft:

1. The `grounding_agent_uitars.py` issue was undercounted. The real requirement is to remove **all remaining bare `eval()` parsing sites**, not only the originally cited subset.
2. The `shell=True` writeup was too broad. The **critical remaining issue is `docker_execute()`**. The local Python execution path in `_execute()` is already argv-based.
3. Actionable fixes, false positives, and duplicates are now separated more clearly.
4. Verification now checks actual closure conditions instead of relying on stale line anchors.

## Phase 0: P0 fixes

### P0-1: Replace all remaining bare `eval()` parsing with a validated literal parser

**File:** `src/khoj/processor/operator/grounding_agent_uitars.py`

**Problem:**  
The file still uses bare `eval(...)` to parse LLM-produced coordinate payloads. The original draft described this as eight inline fixes, but the safer and audit-clean requirement is broader: **no remaining bare `eval()` should be used to parse coordinate or box literals anywhere in this file**.

**Required implementation:**
Create one helper and route every coordinate/box parse through it.

```python
import ast
from typing import Iterable

def _parse_box_literal(raw: str) -> tuple[float, float, float, float]:
    value = ast.literal_eval(raw)

    if not isinstance(value, (list, tuple)) or len(value) != 4:
        raise ValueError(f"Invalid box literal: {raw}")

    parsed = tuple(float(v) for v in value)
    return parsed  # x1, y1, x2, y2
```

Then replace every remaining parse site with:

```python
x1, y1, x2, y2 = _parse_box_literal(start_box)
```

**Why this is better than the original draft:**  
Inline `ast.literal_eval(...)` replacement is directionally correct, but a single helper gives you one place to enforce:
- type checking,
- length checking,
- numeric coercion,
- consistent error messages,
- future telemetry or debug logging if needed.

**Audit closure condition:**  
`grep -n "[^a-zA-Z]eval(" src/khoj/processor/operator/grounding_agent_uitars.py` must return **zero** matches. `ast.literal_eval(` is acceptable.

### P0-2: Remove `shell=True` from Docker Python execution and keep the shell boundary explicit

**File:** `src/khoj/processor/operator/operator_environment_computer.py`

**Problem:**  
The original spec overstated this issue. The **local Python execution path in `_execute()` is already safe** because it calls `subprocess.run(["python3", "-c", python_command_str], ...)`. The critical remaining issue is `docker_execute()`, which still builds an interpolated shell string and executes it with `shell=True`.

**Required implementation for `docker_execute()`:**

```python
async def docker_execute(self, python_command_str: str) -> Optional[str]:
    if not self.docker_container_name or not self.docker_display:
        logger.error("Container name or Docker display not set for Docker execution.")
        return None

    cmd = [
        "docker", "exec",
        "-e", f"DISPLAY={self.docker_display}",
        self.docker_container_name,
        "python3", "-c", python_command_str,
    ]

    process = await asyncio.to_thread(
        subprocess.run,
        cmd,
        shell=False,
        capture_output=True,
        text=True,
        check=False,
    )
```

**Do not do this:**  
Do **not** try to force `shell=False` onto `_execute_shell_command()` wholesale. That method intentionally backs:
- `terminal` commands,
- text-editor `head | tail`,
- shell redirection,
- `sed` replacements,
- Docker `bash -c` command execution.

Those actions are product scope, not accidental shell use.

**Additional hardening required in the same change:**
- Add a short docstring or comment that `_execute()` / `docker_execute()` are the only allowed Python-code execution paths.
- Keep a timeout on Docker execution.
- Log the command boundary in a redacted/structured way without reintroducing interpolation risk.

**Audit closure condition:**  
`docker_execute()` must use an argv list and `shell=False`. `_execute_shell_command()` may stay shell-backed, but its intentionally shell-backed status must be explicit in code comments.

### P0-3: Validate `DB_NAME` before embedded-Postgres SQL construction

**File:** `src/khoj/app/settings.py`

**Problem:**  
The embedded-Postgres setup still interpolates `DB_NAME` into SQL strings. Because `psql()` here is a CLI wrapper, normal query parameterization is not available.

**Required implementation:**
Validate `DB_NAME` immediately after reading it.

```python
import re

DB_NAME = os.getenv("POSTGRES_DB", "khoj")
if not re.match(r"^[A-Za-z_][A-Za-z0-9_]*$", DB_NAME):
    raise ValueError(f"Invalid database name: {DB_NAME}")
```

Only after validation should the code execute:

```python
db_exists_result = PGSERVER_INSTANCE.psql(
    f"SELECT 1 FROM pg_database WHERE datname = '{DB_NAME}';"
)
```

and:

```python
PGSERVER_INSTANCE.psql(f"CREATE DATABASE {DB_NAME};")
```

**Audit closure condition:**  
No unvalidated environment-controlled identifier may be interpolated into the embedded-DB SQL path.

### P0-4: Remove GitHub expression interpolation from shell bodies

**File:** `.github/workflows/run_evals.yml`

**Problem:**  
The original draft correctly identified the class of issue, but the repo still has several `${{ ... }}` expansions embedded directly in shell bodies. The precise line numbers will drift; the rule must be structural.

**Required implementation rule:**  
Inside a `run: |` block, do not embed `${{ ... }}` directly into shell code. Move every expression into `env:` first, then reference only shell variables inside the script.

**Pattern to use:**

```yaml
- name: Example
  env:
    MATRIX_DATASET: ${{ matrix.dataset }}
    HATCH_VERSION: ${{ steps.hatch.outputs.version }}
  run: |
    set -euo pipefail
    echo "Dataset: $MATRIX_DATASET"
    sed -i "s/dynamic = \[\"version\"\]/version = \"$HATCH_VERSION\"/" pyproject.toml
```

**Apply this to at least:**
- the `sed -i ... ${{ steps.hatch.outputs.version }} ...` substitution,
- the eval invocation using `${{ matrix.dataset }}`,
- the summary-generation shell block,
- any other `run:` body still containing `${{`.

**Audit closure condition:**  
A grep for `${{` inside shell bodies in `run_evals.yml` should return zero remaining direct shell-body interpolations.

### P1-2: Eliminate weak secret fallbacks across Django and Starlette

**Files:**
- `src/khoj/app/settings.py`
- `src/khoj/configure.py`

**Problem:**  
The Django `SECRET_KEY` and the Starlette `SessionMiddleware` key still fall back to `!secret`.

**Required implementation:**

In Django settings:

```python
from django.core.exceptions import ImproperlyConfigured

SECRET_KEY = os.getenv("KHOJ_DJANGO_SECRET_KEY")
if not SECRET_KEY:
    raise ImproperlyConfigured("KHOJ_DJANGO_SECRET_KEY must be set")
```

In FastAPI/Starlette middleware setup:

```python
from khoj.app.settings import SECRET_KEY
app.add_middleware(SessionMiddleware, secret_key=SECRET_KEY)
```

**Audit closure condition:**  
There must be one validated secret source and no `!secret` fallback anywhere in the runtime path.

### P1-3: Make cookie policy explicit and OAuth-compatible

**File:** `src/khoj/app/settings.py`

**Problem:**  
The current settings switch to `SameSite=None` on HTTPS. That is broader than necessary. The repo also runs OAuth flows that need redirect compatibility.

**Required implementation:**  
Prefer `SESSION_COOKIE_SAMESITE = "Lax"` and `CSRF_COOKIE_SAMESITE = "Lax"` consistently unless a specific embedded or cross-site POST flow proves `None` is required. Keep `Secure=True` on HTTPS.

If any cookie still truly needs `SameSite=None`, document which cookie and why.

**Audit standard:**  
The code should not carry a broad `None` policy simply because it worked; it should carry the narrowest policy that still preserves login flow.

---

### P1-4: Replace wildcard CORS with actual origin/method/header policy

**File:** `src/khoj/main.py`

**Problem:**  
Current config still uses:
- `allow_methods=["*"]`
- `allow_headers=["*"]`
- invalid `allow_origins` wildcard strings like `http://localhost:*`

**Required implementation:**
- Keep exact shipped app origins in `allow_origins`
- Use `allow_origin_regex` for localhost/127.0.0.1 port variance
- Restrict methods and headers to the actual client/API surface

**Recommended shape:**

```python
CORS_EXACT_ORIGINS = [
    "app://obsidian.md",
    "capacitor://localhost",
    "http://localhost",
    "app://khoj.dev",
    f"{scheme}://{KHOJ_DOMAIN.strip()}",
]

CORS_ORIGIN_REGEX = r"^https?://(localhost|127\.0\.0\.1)(:\d+)?$"
```

with:
- `allow_methods=["GET", "POST", "PUT", "PATCH", "DELETE", "OPTIONS"]`
- `allow_headers=["Authorization", "Content-Type", "Accept", "X-Requested-With", "X-CSRFToken", "X-Khoj-Client"]`

### P1-5: Enable Jinja autoescape without breaking task emails

**File:** `src/khoj/routers/email.py`

**Problem:**  
The repo still creates a global Jinja environment without autoescape, but `send_task_email()` intentionally injects pre-rendered HTML from markdown.

**Required implementation:**

```python
from jinja2 import Environment, FileSystemLoader, select_autoescape
from markupsafe import Markup

env = Environment(
    loader=FileSystemLoader(static_files),
    autoescape=select_autoescape(["html", "xml"]),
)
```

For task-email markdown HTML:

```python
html_result = markdown_it.MarkdownIt().render(result)
html_content = template.render(
    name=name,
    subject=subject,
    query=query,
    result=Markup(html_result),
)
```

That preserves intended HTML while letting Jinja escape everything else by default.
